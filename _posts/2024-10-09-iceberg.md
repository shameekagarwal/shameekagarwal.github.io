## Introduction

- "table format" - table formats are needed for data lake. they already exist for databases / data warehouses. they easily handle where and how to store data in files for us. it has a metadata layer to store these details. there is a "catalog" layer to store details about the different datasets. also, it has a "processing layer" for our queries
- however, "data lakes" do not have these - it just has the ability to store "structured" and "unstructured data" easily
- so, table formats provide these missing layers in data lake
- also, each of these layers can be easily "swapped" for different implementations based on workload
- "hive" was built on top of map reduce, so that we could interact with sql instead of map reduce jobs written in java
- for this, hive had to determine what a table is - it said that all the files in a folder is a table, all the sub folders are "partitions"
- limitations - performance, missing acid guarantees, stale table statistics, etc
- some issues like consistency, performance, etc were achieved by moving data to data warehouses. issue - duplicating data, more work
- so, new approach - instead of tracking which "folders" form a table, we track which "files" form a table
- this information is stored in metadata effectively, so that unlike hive, file listing operations like files / folders stored in a directory etc need not be done repeatedly. this in turn improves the performance
- "iceberg" is a specification - it is not a piece of software. it just enforces that the tools follow this specification to read / write metadata. iceberg is not a continuously running service. when we have an iceberg table, it only has data and metadata in the form of files in storage. iceberg has libraries in java, python, etc which expose apis to read and write data. to query it, we can use engines like spark, dremio, etc

## Iceberg Architecture

![](/assets/img/iceberg/iceberg-architecture.png)

- when spark / dremio want to query an iceberg table, they need to know that a table exists. they do this via a catalog like "glue"
- these catalogs contains all the tables, and a pointer to the location of the "metadata file"
- "metadata file" - the master file in iceberg. there is always only one current metadata file to describe the table
- every time we change the table, a new metadata file is created - nothing is edited, everything here is considered to be immutable
- what metadata files contain - 
  - unique uuid
  - a location for the data of the table
  - schemas (current and old)
  - partitioning schemes (current and old) - another advantage of iceberg, the ability to change the partitioning scheme, without repartitioning the existing data
  - snapshots (current and old) - allows time travel. each snapshot consists of a "manifest list"
- so, spark / dremio now read this "metadata file" to gather all details, and go to the right "manifest list" based on the snapshot they want to be querying
- "manifest list" - list of "manifest files"
- it contains "statistics" about each of the manifest file that it has - things like partitioning scheme, snapshot it belongs to, etc
- this way, it can now get rid of the manifest files that do not make sense for the current query generated by the query engine
- "manifest files" - an entry for each "data file" it tracks. again, it too has "statistics" to help prune data files that do not make sense for the current query generated by the query engine
- now, we only have to handle a handful of these "data files", i.e. the actual parquet files
- if we have a well partitioned and sorted metadata, it will increase our performance
- finally, we would find data/ and metadata/ folders
- metadata/ would have "metadata file" (json), "manifest list" and "manifest" files (both in avro, since they are good for row based queries)
- writing the data goes in the opposite direction - create data files -> create manifest file -> create manifest list -> create metadata file -> update catalog to point to this new metadata file
- this is what helps us maintain acid transactions - the catalog is never updated to point to the new metadata file if the transaction fails

## Benefits of Iceberg

- "smaller updates" are much faster - in hive etc, we would have to update a whole partition and then swap out the partition
- "time travel" using snapshots
- we do not have to maintain "metrics" of table separately, they are automatically updated in the metadata after every change to the table
- "acid transactions" - all engines rely on the metadata.json pointed to by the catalog, and the catalog pointer is updated only when the transaction succeeds
- improved performance - the pruning of "manifest files" and "data files"
- "schema evolution" - the entire data need not be rewritten when the schema changes
- "partition evolution" - same as above - on changing the partitioning scheme, the old data need not be written
- "hidden partitioning" - in hive, assume we would like to partition by month, but we only have a timestamp column. we would have to generate a new month column which we would like to partition based on, maintain extra data, and specify it in queries as well i.e. just filtering by timestamp would not be enough for partition pruning, we would have to filter based on month as well. with iceberg, all of this is eliminated. iceberg will know that we would like to partition based on a "transformed value" of timestamp. we do not have to track an additional column as an analytics engineer, and we do not have to add the additional filter clause for month, filtering by timestamp would be enough
- since the files of a table need not go to the same folder like in hive, we are not throttled by the limitation of requests per unit time for prefixes in object storage

## Versioning

- there is versioning at table level in iceberg
- but here, we are talking about versioning present at catalog level - nessie is a popular catalog
- it allows "branching" like git - the catalog in the feature branch starts pointing to a different metadata file, while the main branch continues pointing to the main metadata file
- this way, we can make changes without affecting the production consumers, since only the metadata file in the feature branch changes
- this also supports "multi table transactions" - iceberg supports this natively, but we would have to use the `begin transaction` and `end transaction` paradigm, so inside a session. with a catalog like nessie, we could make changes to our feature branch for as many sessions and as many tables as we want, and the changes start reflecting once we merge our branch
- supports "rollback" like git for recovery
- "nessie" is cloud agnostic, open source, etc
